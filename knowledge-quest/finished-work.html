A quick SSH explainer

I feel like my classmates, my teacher, and you probably don't already know what SSH and related tools are, so I'm going to go over some key programs that I generated questions about in my semantic map.

SSH

SSH is a program for remotely accessing another computer's command line securely. You can authenticate using a password or a key file stored on both the remote computer and the computer trying to access it.

SCP

SCP is a program to transfer files over SSH, with the same security.

The SSH protocol

Once you authenticate over SSH, if you use it for a shell, it sets up a TCP connection. Over that, it sends your commands in an encrypted form. It uses OpenSSL for the encryption, and you can configure it to use many different methods (here's a list).

SSH keys

SSH keys are a system for authentication I mentioned earlier. You generate an identification file and a public key file, then copy the public key file over to the host you want to be able to log in to. Then, instead of entering your password, your identity is verified.

A rough outline of the verification process is that first, the host you want to log in to generates a random challenge. It sends it to you, and you encrypt it with your identification file. Then you send the encrypted version back, and the host decrypts it with the public key it has, and verifies it matches. Since only you can generate the properly encrypted version if you have your identification file, and only you have it, it's secure.

There are multiple utilities to generate them. The most common one is ssh-keygen. On Windows, you can also use PuTTY. They take a random source, like /dev/random on Mac and Linux (a stream of random bytes generated from user input), and generate two key files based on that.

Security vulnerabilities

SSH has two major versions: SSH1, and SSH2. SSH1 has multiple security vulnerabilities. In the past, SSH also defaulted to an insecure method of encryption, but it has changed since then.

SSH server

The session

The SSH server is constantly listening for connections on port 22, and when a client tries to connect, it asks for authentication. If it succeeds, it initiates the main connection. The client can ask to forward certain things during this time. Then, either an interactive shell is set up or a single command is executed. Finally, when the client is ready, it can ask the server to close the connection.

The shell

When the client asks for a command, SSH uses the default shell, found in /etc/passwd. A brief outline of how Bash (a common default shell) works is first it parses your command, then it expands it, then executes it, and finally returns the result.

SCP

SCP is based on RCP, but instead of using the insecure RSH (remote shell) protocol, SCP utilizes the SSH protocol. When you tell it to upload a file, it secretly runs an instance of SCP on the remote computer via SSH. Then it pipes the files into the connection, and then the remote SCP instance saves the file.
